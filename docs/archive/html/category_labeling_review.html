Category Labeling Review

2026-07-01 · Assessment of the native-tag taxonomy (map v1) + roadmap to maximally accurate & granular per-market labels

Verdict: the primary-category assignment was done well — the architecture is right and the validation was honest — but we are using maybe a third of what the curation actually produced. The baked artifact (market_native_categories.parquet) stores exactly two columns: condition_id, primary. Meanwhile the curation workflow designed a full two-level taxonomy (~35 subcategories across 11 primaries, each with a written FLB rationale), per-tag notes, and structural/mechanic flags — none of which made it into the per-market table. The biggest wins available are bake-outs of things we already built, not new modeling.

1 · What was done well

DecisionWhy it was right
Native tags over LLM labelsTags are Polymarket-maintained, free, and cover every new market automatically (the LLM classification cost ~$200 and freezes at pull time). The deprecated category field was correctly rejected (0.3% populated).
IDF-weighted soft voteEmpirically beat the hard one-primary-per-tag vote (98.7/94.2 vs 97.1/89.2 held-out) — and the comparison was actually run, not assumed.
Structural-tag exclusionSeparating mechanism flags (Up or Down, 5M, Neg Risk, Rewards Automation…) from subject tags is the single most important curation move; it prevented cadence flags from hijacking categories.
Honest, adversarial validationHeld-out 70/30 split against LLM labels (not circular), then two rounds of market-level hand-checks (500-market samples, adversarial subagents): 1.3% error → precedence fix → 0.3%, with the residual identified as genuine ambiguity rather than bugs.
Baked & persisted (v1)The final map + per-market assignments now live on /mnt/data/learnability/native/ (1.42M markets) instead of volatile /tmp.

2 · Where it falls short

2.1 Granularity was thrown away at bake time (the big one)

The curation produced, per tag: primary, subcategory, type (category / structural / ambiguous), and a rationale note. A separate design pass (subcats.json) organized these into ~35 FLB-motivated subcategories, e.g.:

PrimaryDesigned subcategories (each with an FLB rationale)
SportsTeam-sport game markets (by league) · Individual head-to-head (tennis/UFC/boxing/chess) · Field-winner (golf/F1 — the mechanically-strongest-FLB structure) · Olympics/rare events · Props/drafts/fantasy
CryptoCoin price targets (BTC/ETH/SOL…) · Memecoins (no fundamental anchor) · Listings/launches/FDV
FinanceSingle-name equity strikes · Index & commodity (institutionally arbitraged) · Up/down oracle dailies · Earnings/corporate events
PoliticsElections · Legislation/gov action · Appointments (negRisk insider fields) · Scandals/legal one-offs · Approval-metric thresholds
GeopoliticsKinetic strikes · Ceasefire/diplomacy · Trade war/sanctions · Regime stability

None of this exists per-market. The baked table keeps only the primary. These subcategories are exactly the grain the learnability study needs (they separate resolution mechanism and anchorability within topic), and assigning them requires zero new data — the same IDF vote restricted to the winning primary's tags.

2.2 Axis-mixing in the primary column — with a real analytic consequence

The primary set mixes three different kinds of thing: topics (Politics, Sports…), a market mechanic (Mentions), and an event family (Iran). The precedence overrides force any %Iran%-tagged market into 'Iran' — which means the 'Geopolitics' rows of the horizon-matched category table exclude the single biggest geopolitical event family of 2026 (the Israel–Iran war markets). Any 2025→2026 within-Geopolitics comparison is silently composition-shifted. Same structural issue with Mentions: a Trump-SOTU-mentions market is Politics and mentions-mechanic; today it is only 'Mentions'. The fix is to make these separate columns (topic, mechanic, event_family), not siblings in one column.

2.3 No confidence, no abstain, no provenance

The soft-vote margin is computed and discarded. Markets the validators flagged as genuinely ambiguous (health markets with a stray Weather tag, esports events Polymarket tags 'Sports', IPO = Tech-vs-Finance) get a silent forced call, indistinguishable from a market with an unambiguous 20-tag vote. There is also no record of how each market was labeled (vote vs precedence override vs hand fix).

2.4 The curation source file is dirty

Several fixed: true entries in curated.json carry instructions instead of category names in the primary field — e.g. Big Tech → "down-weight vs Finance for valuation questions", Fed/Fed Rates → "Politics in nomination context", Twitter → "lower priority than Politics". A few structural tags also carry non-Other primaries (Depreciated4 → Finance, Breaking News → Politics). These were resolved somewhere on the way into final_tag_map.json, and the end-to-end validation (0.3% market-level error) says the net result is fine — but the source-of-truth chain is muddy and not reproducible from the repo. The persisted final_tag_map_v1.json is currently the only authority.

4 · Roadmap, ranked by value per unit effort

Tier 1 — bake what we already built (one EC2 session, ~1–2 hrs, no new data)
  1. Build market_labels.parquet with the full schema above (minus the ⊕ rows): subcategories, mechanic axis, entities, margins, abstain, provenance, native-meta join. This alone converts the labeling from "primary only" to the most granular per-market description available anywhere for this data.
  2. Hand-lock the top 300–500 markets by dollar volume (one Workflow hand-check → locked_overrides.json). Disagreement concentrates in exactly these markets (94.2% volume-weighted vs 98.7% count-weighted), so a few hundred locks close most of the dollar-weighted error.
  3. Move the taxonomy into the repo — analysis/learnability/tag_taxonomy/ with the versioned map JSON, the subcat design, the builder script, and cleaned curation fields (fix the instruction-in-primary entries explicitly). Right now the only authority is a JSON on /mnt/data built by a scratchpad script.
Tier 2 — close the coverage holes (small cost)
  1. Open-markets re-pull (drop closed="true") → label the censored long-horizon markets; prerequisite for any forward-looking or open-market analysis.
  2. Event-level tag inheritance: markets with no usable tags inherit their parent event's tags before falling to NULL.
  3. Targeted LLM tiebreak on the ambiguous tail (low margin / no tags / conflicts) via Batch API, stored as a separate column — a third opinion, not an overwrite.
Tier 3 — measurement rigor (when writing up)
  1. Per-category precision with CIs: stratified hand-check sample per category (count- and dollar-weighted), reported as a confusion table — treating the LLM labels as a noisy comparator, not truth.
  2. Deterministic question-text cross-check (regexes: "say ___" → mentions, "reach $" → price-target, "vs." → matchup): disagreements with the tag vote become the review queue for map v3.

5 · Bottom line

Was it done well? Yes — for what it set out to do (a clean primary category per market, validated honestly), it's solid and production-ready. The three genuine defects are: the Iran/Mentions axis-mixing (which measurably distorts the 2026 Geopolitics comparison), the missing abstain/confidence, and the muddy curation provenance.

Is there more we can do? A lot, and cheaply: the subcategory taxonomy, mechanic axis, and native structural fields are all already built or already pulled — they just were never assembled into one per-market table. Tier 1 gets you from "one label per market" to "topic + subcategory + mechanic + event family + entities + structural fields + confidence" in a single EC2 session, and the top-market hand-lock is the highest-leverage accuracy fix because that's precisely where the dollars and the residual disagreement live.
