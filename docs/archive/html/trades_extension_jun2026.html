What happened

  1. Diagnosis: our data hard-stopped at Apr 28 because Polymarket redeployed its exchange — CTF Exchange V2 (0xe11118…996b) + NegRisk CTF Exchange V2 (0xe2222d…0f59) — with a new OrderFilled event our extractor didn't know.
  2. Reverse-engineering: decoded the new event OrderFilled(bytes32 orderHash, address maker, address taker, Side side, uint256 tokenId, uint256 makerAmt, uint256 takerAmt, uint256 fee, bytes32 builder, bytes32 metadata). Confirmed tokenId is the real CTF market token (no wrapping) by matching the in-transaction ConditionalTokens transfer.
  3. Integration: wrote a v2 decoder that emits the new events in our existing 12-column raw_events schema (mapping side → the USDC-leg-as-asset-0 convention), so every downstream stage ran unchanged.
  4. Run: extracted 343.4M events (Apr 28→Jun 22) → concat → dedup → fresh CLOB markets pull (mapped the new June tokens at ~100%) → resolutions → Stage-6 expand (2.10B trades) → rebuilt clean (2.019B, resorted).

Validation

The new data is analytically sound. On June BUY trades, realized win rate tracks price almost exactly across all ten deciles (calibration error within ±0.01) — prices, outcomes, and won are all computed correctly through the new V2 path. June BUY trades are 100% matched to a resolution (123.1M/123.1M). Dedup rates on the new months are the normal 0.4–2.5%.

Open follow-ons (markets metadata)

  • eventSlug is sparse on recent months (June ≈ 0.7%) — the Gamma /events backfill undercovers new markets. This affects the up/down exclusion in the FLB pipeline (which keys on eventSlug); a Gamma re-pull or a slug/question-based filter fixes it.
  • market_resolutions_enriched + LLM dims still end at April — the new June markets lack category / resolutionSource / sportsMarketType / LLM labels, so category-sliced learnability analysis won't cover them until a Gamma re-pull (native_repull.py) + dim rebuild over the new condition_ids.
